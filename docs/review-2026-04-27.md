3. **Attachment shape drift (`README.md:194-216` vs `schema/conversation.schema.json:189-201`)**:
   README table fields are `type`, `name`, `mime`, `size_bytes`, `url`, `data_base64`,

   `mime_type`, `size_bytes`, `url`, `data`, `sha256` (required: `id`, `mime_type`). Five

4. **`branches` semantics contradiction** between `README.md:88` ("Leaf message IDs of
   non-linear branches" — i.e. leaves NOT on canonical path) and `README.md:521` ("array
   of leaf message IDs that are not on the canonical path") vs. `test-vectors/13-deep-branches.uacp.json:18`

   — i.e. the canonical leaf IS in branches[]). Two readings. Vector 13's `branches`
   array also contains `m_a2`, which is not a leaf at all. Pick one definition,
   update both sites.
5. **`tool_results[]` arrow in §Appendix B (`README.md:712`)**: maps `tool_results[].tool_call_id`
   → MCP `toolUseId`, but the canonical UACP spec text deprecates `tool_call_id` in favor
   of `call_id` (`README.md:190`, CHANGELOG `Deprecated`). The Appendix should map
   `call_id`. Otherwise readers learn the wrong name.
6. **§9 Sync Protocol (`README.md:389-422`) is HTTP-shaped** (`GET /sync/delta`, `POST
   /sync/upload`) without saying TLS is required, what the `Authorization` token format
   is, what error envelope looks like, idempotency, rate limits, or how a non-HTTP
   transport (libp2p, gRPC) maps. As written this section is closer to "implementation
   sketch" than spec. Either mark it Informative or tighten it.
7. **§7 Context Injection (`README.md:328-361`)**: the literal string `"[FusionLayer
   Context — UACP v0.1.0]"` is hard-coded into a v0.3 spec. (a) version stale, (b) ties
   the standard to one vendor's brand. Replace with a vendor-neutral marker or move to
   informative appendix.
8. **§13 Conformance Levels (`README.md:495`)** says level lives in
   `metadata.uacp_conformance` (snake-style) but the example one line down uses
   `metadata.uacp.conformance` (dotted). Pick one — vectors 11/12/13/14/15 all use
   `uacp.conformance`/`uacp.test.expect` (dotted), so dotted is the de-facto choice;
   prose should match.
9. **§6 Encryption §6.1 (`README.md:309`)** says `salt` decoded length MUST be `≥ 16

   spec's normative byte minimum.
10. **No author/contact line that survives public flip.** README ends with `"Maintainer:

11. **No table of contents.** A 800-line spec with 15 numbered sections + appendices

12. **`message.id` is optional in schema (`schema/conversation.schema.json:77`) but

    without `id`s. Either make `id` required when `branches` is present, or document the

## 5. Spec security

| Question | Answer |
|---|---|
| Mandates authenticated transport? | **No.** §9 sync uses `Authorization: Bearer` over what is implied to be HTTP, but does not require HTTPS or TLS 1.2+ explicitly. (`README.md:396`) |
| Mandates signed envelopes? | **No.** Envelope is AEAD-encrypted (AES-256-GCM auth tag binds AAD), but there is no signature primitive (Ed25519, JWS) on plaintext or envelope. Anyone with the symmetric key can forge a valid envelope. |
| Replay protection? | **No.** No nonces beyond the GCM IV (which is per-message, not per-session); no timestamps inside the envelope; no monotonic sequence numbers in the sync protocol. A replayed `POST /sync/upload` is undetectable. |
| Crypto pinned vs implementation choice? | **Pinned.** AES-256-GCM, IV 12 bytes, tag 16 bytes, Argon2id `(m=65536, t=3, p=1)`, HKDF-SHA256, info `uacp-key-v1`. Good — interop is guaranteed; bad — no agility for post-quantum or weaker-device profiles. |
| Could a malicious impl produce conformance-passing-but-insecure output? | **Yes, several ways.** |
| | a. Reuse IV across multiple ciphertexts (catastrophic for AES-GCM) — schema cannot detect; harness check #7 only verifies *length*. |
| | b. Emit `aad: ""` and never bind the version — accepted by schema, accepted by spec default. AAD-binding-by-default is good but no on-the-wire signal proves the writer used canonical AAD. |
| | c. Weaken the passphrase: the user picks the passphrase; spec mandates argon2id work factor on the *KDF*, not the passphrase entropy. A 4-character passphrase passes conformance. |
| | d. Schema permits `salt` of 16 base64url chars (~12 decoded bytes) even though spec text says ≥ 16 decoded bytes (`encrypted-envelope.schema.json:46-49` vs `README.md:309`). |
| Schema permits injection-style payloads without warning? | **Partly.** `text` content blocks have no `maxLength`; `code.code` has no length cap; `attachment.url` is `format: uri` (no scheme allowlist — `javascript:` and `file://` validate). `image.url` likewise. No content-rendering guidance ("treat all text as untrusted; do not render HTML in `text` blocks"). Add a §Security Considerations section. |
| HTML/script content vectors | `artifact.type` permits `html`, `react`, `svg` with no normative warning that consumers MUST sandbox these (iframe sandbox / dompurify / etc.). |
| Unbounded fields | Many `text`-typed fields lack `maxLength` (`content_block.text`, `content_block.code`, message string-form `content`). For DoS hardening, add normative ceilings or require implementations to enforce one. |

Recommend a new top-level **§16 Security Considerations** section before public flip.

## 6. v0.4 candidate features

Capped at 5; cited from FL signals.

1. **Sources envelope** — `sources: [{ id, kind: "url"|"pdf"|"youtube"|"doc", uri, title?, citation_anchor? }]` at conversation level, with `citations[].source_id` referencing them. Closes the
   NotebookLM/Perplexity Spaces/Claude Projects/ChatGPT GPTs interop gap. (`UACP-GAPS.md:23`,
   `ROADMAP.md:272`, FL #575/#576)
2. **Pack primitive** — `kind: pack` container that bundles persona + policies +
   guidelines as a single installable asset. Required for one-click install across tools.
   (`UACP-GAPS.md:22`, FL #600 done)
3. **§16 Security Considerations + signed envelopes** — Add Ed25519 detached signatures
   over canonical conversation JCS for non-repudiation; add replay defenses (sequence +
   timestamps) to §9 sync. Closes §5 of this review.
4. **IANA media-type registration** — `application/uacp+json` is currently unregistered.
   `GOVERNANCE.md:92` says "IANA provisional registration is planned before v1.0.0".
   Schedule it for v0.4 to lock in the type before adopters proliferate.
5. **Streaming / partial-message envelope** — Today `status: "in_progress"` exists on
   messages but there is no on-the-wire definition of *how* a streaming UACP segment is
   framed (NDJSON? SSE? chunked transfer?). v0.4 should define a streaming profile so
   live capture can interop.
